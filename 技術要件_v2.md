# 技術要件：壁プロジェクション × マイコン入力 体験システム

> **コンセプト**
> 壁面プロジェクションの映像世界を、実物のセンサーやギミックで操作して進む体験型アトラクション。
> 体験者が"場で何をしたか"で進行し、人数を問わず破綻しない設計。

※設計段階であり確定ではありません。
---

## 1. システム全体像

```
┌──────────────┐        Wi-Fi / MQTT         ┌──────────────────┐
│  ESP32 ×複数  │ ◄──────────────────────► │  管理PC（Unity）   │
│  (センサー/LED) │    MQTTブローカ(PC上)経由    │  (映像/音/進行管理) │
└──────────────┘                             └──────────────────┘
```

### 1-1. 役割分担

| レイヤー | 担当 | 責務 |
|---------|------|------|
| **センサー／出力** | ESP32（各ノード） | センサー読み取り、デバウンス処理、LED等のローカル出力、MQTTでのイベント送信 |
| **判定／演出** | 管理PC（Unity） | ステージ進行管理、正解判定、タイムアウト、やり直し、全体リセット、映像・音響演出、ログ保存 |
| **通信中継** | MQTTブローカ（PC上） | ESP32 ↔ Unity 間のメッセージ中継（配送センター役） |

> **ポイント**: ESP32 と Unity は **直接通信しない**。必ずブローカを経由する。

## 2. 通信設計（MQTT）

### 2-1. トピック体系

```
yugo/{stage}/{node}/{種別}
```

| 種別 | 方向 | 用途 | 例 |
|------|------|------|----|
| `evt` | ESP32 → Unity | 入力イベント（タッチ/衝撃/接近 等） | `yugo/stage1/nodeA/evt` |
| `cmd` | Unity → ESP32 | 命令（LED点灯/リセット 等） | `yugo/stage1/nodeA/cmd` |
| `hb`  | ESP32 → Unity | 心拍（死活監視、目安1秒間隔） | `yugo/stage1/nodeA/hb` |

> **node とは**
> 1台の ESP32 マイコン（＋それに繋がったセンサー/LED のまとまり）のこと。
> 同じステージに ESP32 が複数台ある場合に、どのマイコンからのメッセージかを区別するために使う。
>
> 例：ステージ1に ESP32 が2台ある場合
> ```
> yugo/stage1/nodeA/evt  ← タッチパネル担当
> yugo/stage1/nodeB/evt  ← LED担当
> ```

### 2-2. メッセージ形式（JSON）

**evt（入力イベント）**
```json
{ "t": "touch", "id": "fish", "v": 1, "ts": 1700000000 }
```
| フィールド | 意味 |
|-----------|------|
| `t` | イベント種別（`touch` / `hit` / `presence` / `rfid` / `light`） |
| `id` | 対象物の識別子 |
| `v` | 値（1=ON, 0=OFF, またはピーク値等） |
| `ts` | UNIX タイムスタンプ |

**cmd（命令）**
```json
{ "t": "led", "id": "tree", "v": 1 }
```
| フィールド | 意味 |
|-----------|------|
| `t` | 命令種別（`led` / `reset` / `calib`） |
| `id` | 対象物の識別子 |
| `v` | 値 |

### 2-3. データフロー図

```
── 入力イベント（evt）──────────────────────────────────
  ESP32: センサー検知
    │  publish ─► yugo/stage1/nodeA/evt
    ▼
  MQTTブローカ: 中継
    │
    ▼
  Unity: subscribe で受信 → 状態機械を進める → 映像・音を再生

── 命令（cmd）──────────────────────────────────────────
  Unity: 「LED点灯」等を決定
    │  publish ─► yugo/stage1/nodeA/cmd
    ▼
  MQTTブローカ: 中継
    │
    ▼
  ESP32: subscribe で受信 → LED/モータ等を制御

── 心拍（hb）──────────────────────────────────────────
  ESP32: 1秒ごとに publish ─► yugo/stage1/nodeA/hb
    │
    ▼
  Unity: subscribe で監視 → 5秒途絶 → スタッフ警告 / フェイルセーフ
```

## 3. 運用要件

| 項目 | 仕様 | 備考 |
|------|------|------|
| スタッフ用リセット | ボタン1つで全ステージ初期化 | 物理ボタン or 管理画面 |
| 自動リセット | 無操作 N 分で初期状態に戻る | 既定 2分（現場で調整可） |
| デバウンス | 50ms | チャタリング防止 |
| クールダウン | 300ms | 連打・誤検知防止 |
| Watchdog | ESP32 内蔵 WDT | フリーズ時に自動再起動 |
| 心拍監視 | MQTT `hb` トピック（1秒間隔 / 5秒タイムアウト） | 通信断を検知 |
| ログ | 時刻・ステージ・ノード・内容を管理PCに保存 | 障害解析・改善用 |

## 4. ステージ仕様

各ステージは **入力 → 成立条件 → 出力 → 通信 → リセット** の5項目で定義する。

---

### ステージ1：遺跡の入口 — 生き物の順番で道を開く

> 参照：`image/企画書_ページ_08.jpg`

```
体験者 ──► [魚/兎/鹿を順番にタッチ] ──► 木が開いて道が出る
```

| 項目 | 内容 |
|------|------|
| **入力（候補）** | 静電タッチ × 3面（魚・兎・鹿）/押しボタン × 3 / タッチディスプレイ / ピエゾ |
| **成立条件** | `fish → rabbit → deer` の順に入力（順序・制限時間・リトライは Unity で管理） |
| **出力（物理）** | 木アイコン LED 点灯（成功時） |
| **出力（映像）** | 木が開く → 道が出現 → 次ステージへ遷移 |
| evt | `{ "t":"touch", "id":"fish\|rabbit\|deer", "v":1 }` |
| cmd | `{ "t":"led", "id":"tree", "v":1 }` / `{ "t":"reset", "v":1 }` |
| **リセット** | 自動：無操作 N 分 or 入力途中タイムアウト / 手動：スタッフリセット |

---

### ステージ2：地下都市 — 3つのミニステージ

> 参照：`image/企画書_ページ_11.jpg`

#### 2-1. 質問（住民に聞く）

```
体験者 ──► [NPCの前に立つ] ──► NPC音声再生 + ヒント（暗証番号の一部）
```

| 項目 | 内容 |
|------|------|
| **入力（候補）** | 距離センサー or 赤外線（来訪検知）/ 呼び鈴/ボタン/受話器 / 音声処理 / 画像処理 / 床マット（ピエゾ）|
| **成立条件** | 検知が 1秒以上 継続 |
| **出力（映像/音）** | NPC音声再生 + ヒント表示 |
| evt | `{ "t":"presence", "id":"npc", "v":1 }` |

#### 2-2. 掃除（こすって文字を出す）

```
体験者 ──► [壁面をこする] ──► 汚れが落ちて隠し文字が露出
```

| 項目 | 内容 |
|------|------|
| **入力（候補）** | 面タッチ（銅箔/導電布の格子 + 静電タッチIC）/パネル数点の圧力センサー（FSR/ピエゾ） / ブラシに加速度センサー|
| **成立条件** | こすり量（面積 or 回数）の累積が閾値に到達 |
| **出力（映像）** | 汚れが徐々に落ち、隠し文字が露出 |
| evt | `{ "t":"touch", "id":"clean", "v":1 }` （周期送信 or 閾値超え時のみ） |

#### 2-3. 瓦礫（ピッケルで壊す）

```
体験者 ──► [ピッケルで瓦礫を叩く] ──► 瓦礫崩壊 → 通路解放
```

| 項目 | 内容 |
|------|------|
| **入力（候補）** | 瓦礫面のピエゾ（衝撃検知） |
| 入力（強化案） | ピッケル側 IMU + 瓦礫ピエゾの AND 条件（誤検知低減） |
| **成立条件** | 規定回数/パワーの打撃成立（既定：強打 8回） |
| **出力（映像）** | 瓦礫崩壊 → 通路解放 |
| evt | `{ "t":"hit", "id":"rubble", "v":<peak値> }` |

**ステージ2 共通リセット**: 自動（無操作 2分） / 手動（スタッフリセット）

---

### ステージ3：崩壊した空間 — 岩を集めて台座に揃える

> 参照：`image/企画書_ページ_15.jpg`

#### 3-1. 岩/洞窟（ライトで発光させて取得）

```
体験者 ──► [専用ライトを岩に照射] ──► 岩が光る → 取得
```

| 項目 | 内容 |
|------|------|
| **入力（候補）** | フォトダイオード + 専用ライトの点滅パターン検出（外光耐性◎）/照度センサー閾値判定（外光の影響大、要調整） |
| **成立条件** | 照射 1秒継続 → （任意）取得ボタン押下 |
| **出力（物理）** | 対象物 LED 発光 |
| **出力（映像）** | 岩が光る / 取得演出 |
| evt | `{ "t":"light", "id":"cave", "v":1 }` / `{ "t":"touch", "id":"pickup", "v":1 }` |

#### 3-2. 台座（3つの岩を揃える）

```
体験者 ──► [3種の岩を台座に置く] ──► 解放演出
```

| 項目 | 内容 |
|------|------|
| **入力（候補）** | RFID/NFC（岩3種にタグ、台座にリーダー）/磁石 + ホールセンサー（置いた判定のみ、種別判別不可） |
| **成立条件** | 3種の岩（赤/緑/青）が すべて 台座に揃う |
| **出力（物理）** | 台座 LED（達成状態表示） |
| **出力（映像）** | 解放演出 |
| evt | `{ "t":"rfid", "id":"rock_red\|rock_green\|rock_blue", "v":1 }` |

**ステージ3 共通リセット**: 自動（無操作 N 分） / 手動（スタッフリセット）

## 6. 技術スタック（想定）

| 領域 | 技術 |
|------|------|
| 映像/進行管理 | Unity 6.0（6000.0.68f1 LTS）/ C# / .NET 8 |
| マイコン | ESP32（Arduino or ESP-IDF） |
| 通信 | MQTT over Wi-Fi（ブローカ: Mosquitto 等） |
| **Unity 側 MQTT クライアント** | **M2Mqtt**（動作確認済み） |
| センサー類 | 静電タッチIC / ピエゾ / 距離センサー / RFID-RC522 / フォトダイオード |
| 出力類 | LED / プロジェクター |

### Unity 用 MQTT クライアント比較

| ライブラリ | Unity 相性 | メンテ状況 | 特徴 |
|-----------|-----------|-----------|------|
| **M2Mqtt** ✅ 決定 | ◎ | 安定（枯れている） | 軽量・シンプルな API。Unity 6.0 LTS (6000.0.68f1) で動作確認済み |
| **MQTTnet 4.x** | ◎ | 活発 | .NET 8 ネイティブ対応・async/await ◎。Unity 6 なら制約なく使える |
| **mqtt-unity** | ◎ Unity 専用 | 非公式ラッパー | M2Mqtt を Unity 向けに使いやすくしたもの |

> **.NET 互換性について**
> Unity 6 は .NET 8 を採用しているが、.NET 8 は .NET Standard 2.0 と後方互換があるため M2Mqtt も引き続き動作する。
> ```
> .NET 8  ⊃  .NET Standard 2.1  ⊃  .NET Standard 2.0  ← M2Mqtt はここ
> ```
> つまり Unity 6 にしても選択肢は減らない。まず M2Mqtt で動作確認し、必要なら MQTTnet に移行という流れが取りやすい。

> **Unity のスレッド注意点**: MQTT の受信コールバックはワーカースレッドで呼ばれるため、Unity の GameObject 操作はメインスレッドに戻す必要がある。`ConcurrentQueue` などで対処する。
