# 全体システム（壁プロジェクション + マイコン入力）
体験をステージごとに分割し、各ステージは **入力（センサー）→判定（状態）→出力（演出/物理）** に統一する。
進行は「誰が操作したか」ではなく「場で何が成立したか」で扱い、1人でも複数人でも破綻しない設計にする。

## 役割分担
- 管理PC（Unity）：進捗、正解、タイムアウト、やり直し、全体リセット、ログ、演出（映像/音）を管理
- マイコン（ESP32）：センサー読み取り、デバウンス、ローカル出力（LED等）、管理PCへイベント送信

## 通信
- 方式：ESP32（Wi‑Fi）→ 管理PC（MQTTブローカ）
- 目的：イベント駆動（入力・成立・心拍）を統一し、ステージ追加に強くする

## データの流れ（ESP32 ↔ MQTTブローカ ↔ Unity）
「MQTTブローカ」は配送センター、ESP32とUnityは“送り手/受け手”になる。
ESP32とUnityは **直接つながらず**、必ずブローカ経由でやり取りする。

```
[ESP32(センサー)] --Wi‑Fi--> [MQTTブローカ(PC上)] <--> [Unity(演出/進行)]
   publish(evt/hb)             配達（中継）            subscribe(evt/hb)
   subscribe(cmd)                                   publish(cmd)
```

### 1) ESP32 → Unity（入力イベント：evt）
1. ESP32がセンサーを検知
2. ESP32が `.../evt` にpublish（例：touch/hit/presence/rfid/light）
3. Unityは `.../evt` をsubscribeして受信
4. Unityが状態機械を進め、映像/音を再生

例：
- topic：`yugo/stage1/nodeA/evt`
- payload：`{ "t": "touch", "id": "fish", "v": 1, "ts": 1700000000 }`

### 2) Unity → ESP32（命令：cmd）
1. Unityが「LED点灯」「リセット」などを決定
2. Unityが `.../cmd` にpublish
3. ESP32は `.../cmd` をsubscribeして受信
4. ESP32がLED/モータ等の出力を実行

例：
- topic：`yugo/stage1/nodeA/cmd`
- payload：`{ "t": "led", "id": "tree", "v": 1 }`

### 3) ESP32 → Unity（心拍：hb）
1. ESP32が一定周期（目安1秒）で `.../hb` をpublish（「生きてる」通知）
2. Unityが `.../hb` をsubscribeして監視
3. 受信が途切れたら（目安5秒）「通信断/フリーズ」と判断し、スタッフ警告・フェイルセーフ動作

### MQTTトピック（例）
- `yugo/{stage}/{node}/evt` : ESP32 → PC（入力イベント）
- `yugo/{stage}/{node}/cmd` : PC → ESP32（点灯/リセット等）
- `yugo/{stage}/{node}/hb` : ESP32 → PC（心拍）

### メッセージ形式（例：JSON）
- evt：`{ "t": "touch|hit|presence|rfid|light", "id": "fish", "v": 1, "ts": 1700000000 }`
- cmd：`{ "t": "led|reset|calib", "id": "tree", "v": 1 }`

## 運用要件（必須）
- スタッフ用リセットボタン（全体）
- 自動リセット：無操作 `N` 分（既定：2分、現場で調整）
- 誤検知対策：デバウンス（既定：50ms）/クールダウン（既定：300ms）
- 死活監視：Watchdog（ESP32）+ 心拍（MQTT）
- ログ：各イベントの時刻・ステージ・ノード・内容を管理PC側で保存

# ステージ仕様（I/O表）
各ステージは「入力」「成立条件」「出力」「通信イベント」「リセット条件」を最低限定義する。

## ステージ1：遺跡の入口
参照画像：`image\企画書_ページ_08.jpg`

### 目的
- 生き物の順番入力で道を開く（映像遷移 + 木アイコン点灯）

### I/O
|項目|内容|
|---|---|
|入力（候補）|静電タッチ（魚/兎/鹿の3面）|
|入力（候補）|押しボタン（3個）|
|入力（候補）|タッチディスプレイ|
|入力（候補）|ピエゾ|
|成立条件|`fish → rabbit → deer` の順で入力（順序、制限時間、リトライをUnity側で管理）|
|出力（物理）|木アイコンLED点灯（成功時）|
|出力（映像）|木が開く/道が出る演出 → 次ステージへ|
|evt（例）|`touch id=fish|rabbit|deer v=1`|
|cmd（例）|`led id=tree v=1`、`reset v=1`|

### リセット
- 自動：無操作N分 or 入力途中でタイムアウト
- 手動：スタッフリセット

## ステージ2：地下都市（質問/掃除/瓦礫）
参照画像：`image\企画書_ページ_11.jpg`

### 2-1 質問（住民に聞く）
|項目|内容|
|---|---|
|入力（候補）|来訪検知（距離センサー or 赤外線）|
|入力（候補）|呼び鈴/ボタン/受話器|
|入力（候補）|音声処理、画像処理|
|入力（候補）|床マット（ピエゾ）|
|成立条件|検知が一定時間継続（既定：1秒）|
|出力（映像/音）|NPC音声再生 + ヒント表示（暗証番号の一部）|
|evt（例）|`presence id=npc v=1`|

### 2-2 掃除（こすって文字を出す）
|項目|内容|
|---|---|
|入力（候補）|面タッチ（銅箔/導電布の格子 + 静電タッチIC）|
|入力（候補）|パネル数点の圧力センサー（FSR/ピエゾ）|
|入力（候補）|ブラシに加速度センサ|
|成立条件|「こすり量」累積が閾値到達（一定面樹、回数）|
|出力（映像）|汚れが落ちて隠し文字が露出|
|evt（例）|`touch id=clean v=1`（一定周期で送る or しきい値越えのみ送る）|

### 2-3 瓦礫（ピッケルで壊す）
|項目|内容|
|---|---|
|入力（候補）|瓦礫面のピエゾ（衝撃）|
|入力（強化案）|ピッケル側IMU + 瓦礫ピエゾのAND条件（誤検知低減）|
|成立条件|規定回数/パワーの「破壊」成立（既定：強打8回）|
|出力（映像）|瓦礫崩壊 → 通路解放|
|evt（例）|`hit id=rubble v=peak`|

### リセット
- 自動：無操作N分（既定2分）
- 手動：スタッフリセット

## ステージ3：崩壊した空間（台座/岩/洞窟）
参照画像：`image\企画書_ページ_15.jpg`

### 3-1 台座（3つの岩を揃える）
|項目|内容|
|---|---|
|入力（候補）|RFID/NFC（岩3種にタグ、台座にリーダ）|
|入力（候補）|磁石 + ホールセンサー（置いた判定のみ）|
|成立条件|3種の岩が台座に揃う|
|出力（物理）|台座LED（達成状態表示）|
|出力（映像）|解放演出|
|evt（例）|`rfid id=rock_red|rock_green|rock_blue v=1`|

### 3-2 岩/洞窟（ライト照射で発光→取得）
|項目|内容|
|---|---|
|入力（候補）|受光（フォトダイオード）+ 専用ライトの点滅パターン検出（外光に強い）|
|入力（候補）|照度センサー閾値判定（外光の影響が大きいので要調整）|
|成立条件|照射成立（既定：1秒継続）→ 取得ボタン（任意）|
|出力（物理）|対象物LED発光|
|出力（映像）|岩が光る/取得演出|
|evt（例）|`light id=cave v=1`、（任意）`touch id=pickup v=1`|

### リセット
- 自動：無操作N分
- 手動：スタッフリセット
